Mismatch Web App:

Code | Site

Have you ever wanted to know who follows you on Instagram that you don't follow back?
What about the accounts you follow that don't return the favor?
Mismatch will help you find out. Visitors log into this browser-based web app using their Instagram accounts. It communicates with the Instagram API to get a list of the visitor's followers as well as the accounts that the user follows. This information is passed through a node server before being graphically displayed using CSS. Once logged in, the visitor can search other handles to find information on the specified user.


CHALLENGE: Hide the user's access token.

SOLUTION: When the app first went live, there was no server and the access token was passed to the app attached to the redirect URL. To keep this from being shared by anyone passing the link along, I stored the token as a JavaScript variable and cut the token out of the URL window.
var here = window.location.href;
var token = here.slice(here.indexOf('access_token=')+13);

CHALLENGE: Find and display the difference between mismatches efficiently.

SOLUTION: When I first built the page, I was making calls to the Instagram API to populate two arrays; one for the user's followers and the other for the accounts that the user follows(ATTUF). Then I would make two more calls to the API, just to get the usernames for each list, in order to compare them and list them on the side of the page. This resulted in hundreds of calls to the Instagram API, which ended up going over my hourly allowance. This taught me that there was a limit to how many calls I could make.

To do this more efficiently, I created an array for the user's followers as well as the ATTUF and added them to the user's data object. These two arrays get passed to function oneOrderedArray(), which takes each connection's object and adds following, followedBy or both as keys, sets them as true or false and returns a single array of connections sorted into alphabetical order. This array is fed into the display() function, which checks those keys to see how each user should show up on the site.

At first, I printed the mismatches in two lists next to the section displaying followers and ATTUF. That wasn't aesthetically pleasing, so I removed the list, choosing to graphically mark the mismatching accounts in the display section instead. I thought it could still be improved, so I consolidated down to one master list of all of the users associated with the searched account and used CSS to visualize the relationship between the user and each connection.

CHALLENGE: When the visitor searched for a different handle, their connections were still showing up under the searched user's heading.

SOLUTION: I found the bug by logging the search results to the console when a new handle was typed in. This showed me that the search was returning that handle's information, it just wasn't displaying properly. That led me to look to the bottom of the list, where I found the new search's connections appended. It was an easy fix to make sure the old information was cleared every time a new search was performed and with every other addition I've remembered to clear what was there before.

@AnotherCoat Twitter Bot:

Code | Site

When I decided that I wanted to create a Twitter bot, I wasn't sure what it should tweet. I ended up deciding to do nail polish colors because some brand give them fun names! While looking for the information to populate the data file, I learned about scraping web pages and how to properly format data to be traversed later. This Twitter bot randomly picks a nail polish color from a JSON file and tweets the brand, color name and which collection that color is from.


CHALLENGE: Scrape the internet for full lists of color names.

SOLUTION: I Googled around to find pages with comprehensive lists of colors for three popular nail polish brands: OPI, Essie and Butter London. Once I found what I was looking for, I inspected the HTML to see how the data was formatted. I scraped the sites to create an array of HTML items by typing var brand = document.getElementsByClassName('class') into the console, where brand and class were substituted according to each site.

CHALLENGE: Format data effectively to be randomly grabbed and tweeted.

SOLUTION: Once I had an array of HTML elements, I would write a function using a for loop to grab the brand, color and collection from each item, which I then added to a JavaScript object. I ended up with a JSON formatted to have collections stored as keys whose values were objects containing each color as a key with the brand as it's value. Ex:
{
"Sheers": {
"Blanc": "Essie",
"Private Weekend": "Essie",
"Marshmallow": "Essie",
"Waltz": "Essie"
}
};

When the JSON file was complete, I wrote a randomizer() function to select a collection, from which it would randomly select a color, which my generateTweet() function would use to compose a string to send as a tweet.
function generateTweet(obj){
var collection = randomizer(obj);
var color = randomizer(obj[collection]);
var brand = obj[collection][color];
var tweet = color + ' by ' + brand + ' from the ' + collection + ' Collection.'
if (tweet.length <= 140){
return tweet;
}
else{
generateTweet(obj);
}
};

Tabber Chrome Extension:

Code

The Tabber Chrome extension tracks when tabs are opened and closed and displays that information in a popup. The extension offers the user the option to view a D3.js graph, which opens in a new tab, to view hourly averages. The information is stored in Chrome's Local Storage and can be downloaded as a JSON or a TSV from the popup.


CHALLENGE: Store and format tab open and close data to be displayed in a D3 graph.

SOLUTION: Whenever a tab is opened or closed, the extension's background.js file creates an object with three keys: Date, Opened and Closed. The Date key gets a timestamp as its value. If a tab was opened, the value of Opened is set to 1 and Closed is set to 0, with the reverse happening when a tab closes. This object is pushed to the Hourly array, which is kept in Chrome's Local Storage. This array is used to find the average hourly opens and closes for the D3.js graph.

CHALLENGE: Keep data through extension reloads.

SOLUTION: Throughout its creation, I had to reboot the extension many times. Whenever I did, the data I had stored in Chrome's Local Storage would be wiped. As I was watching my data during the refresh, I noticed that the previous information would only disappear once new data would overwrite it. This led me to my solution:
if (getStorage('hourly') !== undefined){
hourly = getStorage('hourly');
};

This if statement allowed me to set the variable equal to what it was before the extension refreshed, keeping the data through each new update.

CHALLENGE: Make the collected data downloadable.

SOLUTION: I decided to make the collected data available for download as both a JSON file and a TSV file. The JSON gives more data, including every time a tab was opened or closed since the extension started running, while the TSV gives the hourly averages that get graphed. When the user clicks one of the buttons to download data, a call is made to Local Storage to get either the data stored in tracker (for the JSON) or the data stored in hourly (for the TSV). In order for the information to download properly, I encoded the data as a base-64 string.
base64 = window.btoa(localStorage.getItem('tracker'));
var url = 'data:application/octet-stream;base64,' + base64;
chrome.downloads.download({
url: url,
filename: 'tab_activity.json'
});

Tic-Tac-Toe

Code | Site

This is the first project I built from scratch for my application to Hacker School Recurse Center. It is a two-player Tic-Tac-Toe game that tracks how many wins each player has and alternates which player starts each game.


CHALLENGE: Check for the winner efficiently each time a move is made.

SOLUTION: Each tile has an id number. When the user clicks a tile that is available, a function called setMove() pushes an object containing the currentPlayer and the tile's id to an array called boxes. Each time setMove() is called, function checkWin() gets called as well. This runs function getPlayerMoves(), which looks at boxes and returns an array of the current player's tiles. This array is checked against all possible winning combinations. If there is a match, the setWin() function is called, which turns off pointer events for all tiles using CSS.

CHALLENGE: Track the amount of wins each player has.

SOLUTION: I used a JavaScript Object called tally to store the amount of wins each player has. When a visitor first arrives at the page, tally.forX and tally.forO are both set to zero. Once the checkWin() function returns true, the setWin() function gets called with the currentPlayer passed in as a parameter. This function increases the appropriate player's tally based on the input.

CHALLENGE: Alternate player turn in the game and player starting each game.

SOLUTION: I created a JavaScript variable currentPlayer, which is originally set to "X", who always starts the first game. I also created a function called otherPlayer(), which which takes a player as a parameter and then returns the opponent.
function otherPlayer(player){
var mapping = {"X": "O", "O": "X"};
return mapping[player];
};

Each time a player moves, currentPlayer is set as what otherPlayer(currentPlayer) returns. Once the game is over, the resetBoard() function is called, which figures out whose turn it is by adding tally.forX and tally.forO. If the number is even, X will start the next game. If it's odd, O starts. This means that in the case of a draw, nobody wins and the same player starts the next game.
